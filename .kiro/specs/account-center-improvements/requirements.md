# 需求文档：账号中心界面改进

## 1. 概述

### 1.1 功能描述
改进账号中心的账号卡片展示和交互功能，包括响应式布局优化、账号快速切换、反代配置管理和额度信息细化展示。

### 1.2 目标用户
使用 ProxyForge 管理多个 Kiro/AWS Builder ID 账号的用户。

### 1.3 业务价值
- 提升多账号管理效率
- 改善大屏幕设备的空间利用率
- 提供更直观的账号状态和额度信息
- 支持账号级别的反代配置

## 2. 功能需求

### 2.1 响应式卡片布局优化

**需求ID**: REQ-001  
**优先级**: 高

#### 描述
账号卡片应该支持自动填充布局，每个卡片固定宽度 280px，根据容器宽度自动计算每行能放置多少个卡片，实现真正的响应式布局。

#### 验收标准
- [ ] 每个卡片固定宽度为 280px
- [ ] 根据容器宽度自动计算每行卡片数量（能放几个算几个）
- [ ] 卡片之间的间距保持一致（12px）
- [ ] 当容器宽度变化时，卡片自动换行
- [ ] 最后一行卡片左对齐，不拉伸填充
- [ ] 窗口大小调整时布局自动重排
- [ ] 保持卡片内容的完整显示，不被截断

#### 布局示例
- 容器宽度 980px（最小窗口）：约 3 个卡片/行
- 容器宽度 1280px：约 4 个卡片/行
- 容器宽度 1920px：约 6 个卡片/行
- 容器宽度 2560px：约 8 个卡片/行

#### 技术约束
- 使用 CSS Grid 的 `auto-fill` 或 `auto-fit` 特性
- 或使用 Flexbox 的自动换行
- 保持现有卡片的内部结构和样式
- 不影响卡片的选中状态视觉效果

---

### 2.2 账号快速切换功能

**需求ID**: REQ-002  
**优先级**: 高

#### 描述
在每个账号卡片上添加"切换账号"按钮，点击后拉起 Kiro 桌面客户端并切换到该账号。当前活跃账号的切换按钮应显示为禁用状态。

#### 验收标准
- [ ] 每个账号卡片右上角显示"切换账号"图标按钮
- [ ] 点击按钮后调用 `window.desktop.automation.launchKiroClient` 接口
- [ ] 切换成功后更新账号的 `kiroClientLastLaunchAt` 时间戳
- [ ] 当前活跃账号的切换按钮显示为灰色/禁用状态
- [ ] 非 Electron 环境下，切换按钮不显示或显示为禁用状态
- [ ] 切换过程中显示加载状态（按钮禁用 + 加载图标）
- [ ] 切换失败时显示错误提示
- [ ] 切换成功后在日志区域显示成功消息

#### 活跃账号判定规则
- 最近30分钟内启动过的账号视为活跃账号
- 如果有多个账号符合条件，选择 `kiroClientLastLaunchAt` 最新的账号
- 如果没有活跃账号，所有切换按钮都可用

#### 技术约束
- 只在 `providerId === 'kiro'` 时显示切换按钮
- 需要检查 `window.desktop?.automation?.launchKiroClient` 是否存在
- 使用现有的 `launchSelectedKiroClient` 函数逻辑
- 切换按钮不应影响卡片的点击选中功能

---

### 2.3 反代配置功能

**需求ID**: REQ-003  
**优先级**: 中

#### 描述
在每个账号卡片上添加"反代"按钮，点击后弹出配置面板，允许用户为该账号配置独立的反代服务器设置。

#### 验收标准
- [ ] 每个账号卡片右上角显示"反代"图标按钮（与切换按钮并列）
- [ ] 点击按钮后弹出模态框或侧边抽屉
- [ ] 配置面板包含以下字段：
  - 启用/禁用开关
  - 反代服务器地址（必填，格式验证）
  - 端口号（必填，1-65535）
  - 协议类型（HTTP/HTTPS/SOCKS5）
  - 认证用户名（可选）
  - 认证密码（可选，密码输入框）
- [ ] 配置保存后更新账号数据并持久化到 localStorage
- [ ] 已配置反代的账号，反代按钮显示不同颜色（如绿色）
- [ ] 配置面板显示当前配置的连接状态（已启用/已禁用）
- [ ] 支持测试连接功能（可选）
- [ ] 关闭配置面板时，未保存的更改应提示用户确认

#### 数据结构
在账号对象中添加 `proxyConfig` 字段：
```javascript
{
  enabled: boolean,
  server: string,
  port: number,
  protocol: 'http' | 'https' | 'socks5',
  username: string,
  password: string
}
```

#### 技术约束
- 配置数据存储在 `accountCenter` store 中
- 使用 Vue 3 的 Teleport 组件实现模态框
- 表单验证使用原生 HTML5 验证或简单的自定义验证
- 反代配置不影响账号的其他功能

### 2.4 额度信息细化展示

**需求ID**: REQ-004  
**优先级**: 高

#### 描述
卡片需要显示更详细的额度信息，包括套餐额度、免费额度和其他福利额度，而不是只显示总额度。

#### 验收标准
- [ ] 卡片显示以下额度信息：
  - 套餐额度（已用/总量）
  - 免费额度（已用/总量）
  - 福利额度（已用/总量）
  - 总额度（已用/总量）
- [ ] 每种额度类型使用不同的颜色标识
- [ ] 额度进度条支持分段显示（不同类型用不同颜色）
- [ ] 当某类额度用尽时，显示警告标识
- [ ] 额度数据来源于 API 拦截或手动同步
- [ ] 支持手动刷新额度信息的按钮
- [ ] 额度信息加载失败时显示"待同步"状态

#### 数据结构扩展
在账号对象中扩展额度字段：
```javascript
{
  quotaUsed: number,           // 总已用
  quotaTotal: number,          // 总额度
  quotaBreakdown: {            // 额度细分（新增）
    subscription: {
      used: number,
      total: number,
      type: 'paid'
    },
    freeTier: {
      used: number,
      total: number,
      type: 'free'
    },
    bonus: {
      used: number,
      total: number,
      type: 'promotional'
    }
  }
}
```

#### 技术约束
- 需要扩展 `electron/main.cjs` 中的 `extractUsageAndModels` 函数
- 需要识别 API 响应中的细分额度字段（如 `subscriptionQuota`, `freeQuota`, `bonusQuota` 等）
- 如果 API 不提供细分数据，则只显示总额度
- 保持向后兼容，旧账号数据不包含 `quotaBreakdown` 时使用总额度显示

### 2.5 API 接口调研

**需求ID**: REQ-005  
**优先级**: 高

#### 描述
调研 Kiro 客户端获取额度信息的 API 接口，确定额度数据的来源和字段结构。

#### 验收标准
- [ ] 识别出获取额度信息的 API 端点
- [ ] 记录 API 请求和响应的完整结构
- [ ] 确定套餐额度、免费额度、福利额度的字段名称
- [ ] 确定额度刷新的频率和触发条件
- [ ] 记录可能的额度类型和状态值
- [ ] 更新 `isInterestingKiroApi` 函数以捕获额度相关的 API
- [ ] 更新 `extractUsageAndModels` 函数以提取细分额度

#### 调研方法
1. 在 `captureKiroWebAccount` 函数中添加更详细的 API 响应日志
2. 手动登录 Kiro 账号，观察网络请求
3. 分析包含 `usage`, `quota`, `credit`, `billing`, `subscription`, `entitlement` 等关键词的 API
4. 记录不同账号状态下的 API 响应差异（新账号、付费账号、免费账号等）

#### 输出文档
创建 `api-research.md` 文档，记录：
- API 端点列表
- 请求参数
- 响应结构示例
- 字段映射关系
- 特殊情况处理

## 3. 非功能需求

### 3.1 性能需求
- 卡片布局变化时的过渡动画应在 300ms 内完成
- 切换账号操作应在 3 秒内完成（不包括客户端启动时间）
- 反代配置保存应在 500ms 内完成
- 额度信息刷新应在 5 秒内完成

### 3.2 兼容性需求
- 支持 Windows、macOS、Linux 平台的 Electron 环境
- 支持最小窗口宽度 980px
- 保持与现有账号数据结构的向后兼容

### 3.3 可用性需求
- 所有交互按钮应有明确的图标和提示文本
- 错误消息应清晰易懂，提供解决建议
- 加载状态应有明确的视觉反馈
- 配置面板应支持键盘导航

### 3.4 安全需求
- 反代配置中的密码应加密存储
- 敏感信息不应出现在日志中
- 配置数据应只存储在本地，不上传到服务器

## 4. 用户故事

### US-001: 大屏幕多列显示
**作为** 使用大屏幕设备的用户  
**我希望** 能在一行看到更多账号卡片  
**以便** 更高效地浏览和管理多个账号

**验收标准**:
- 在 1920px 宽度的屏幕上，一行自动显示约 6 个账号卡片（280px 固定宽度）
- 卡片内容完整显示，不被截断
- 卡片之间的间距合理，视觉上不拥挤
- 窗口缩小时卡片自动换行，不会被挤压变形

---

### US-002: 快速切换账号
**作为** 管理多个 Kiro 账号的用户  
**我希望** 能一键切换到指定账号  
**以便** 快速在不同账号之间工作，无需重复登录

**验收标准**:
- 点击切换按钮后，Kiro 客户端自动启动并切换到该账号
- 当前使用的账号有明确的视觉标识
- 切换失败时有清晰的错误提示

---

### US-003: 配置账号反代
**作为** 需要使用代理的用户  
**我希望** 能为每个账号配置独立的反代设置  
**以便** 在不同网络环境下使用不同的代理策略

**验收标准**:
- 可以为每个账号单独配置反代服务器
- 配置保存后下次打开应用时仍然有效
- 已配置反代的账号有明确的视觉标识

---

### US-004: 查看详细额度
**作为** 关注账号使用情况的用户  
**我希望** 能看到套餐额度、免费额度等详细信息  
**以便** 更好地规划账号使用策略，避免超额

**验收标准**:
- 卡片清晰显示各类额度的使用情况

## 5. 技术实现建议

### 5.1 响应式布局
使用 CSS Grid 的 `auto-fill` 实现固定宽度自动填充：
```html
<!-- 方案1: CSS Grid -->
<div class="grid gap-3" style="grid-template-columns: repeat(auto-fill, minmax(280px, 280px));">
  <!-- 卡片 -->
</div>

<!-- 方案2: Tailwind CSS 自定义类 -->
<div class="grid gap-3 grid-cols-[repeat(auto-fill,minmax(280px,280px))]">
  <!-- 卡片 -->
</div>

<!-- 方案3: Flexbox -->
<div class="flex flex-wrap gap-3">
  <div class="w-[280px]"><!-- 卡片 --></div>
</div>
```

### 5.2 切换按钮实现
在卡片右上角添加按钮组：
```vue
<div class="absolute top-3 right-3 flex gap-2">
  <button @click.stop="switchAccount(item)" :disabled="isActiveAccount(item)">
    <SwitchIcon />
  </button>
  <button @click.stop="openProxyConfig(item)">
    <ProxyIcon />
  </button>
</div>
```

### 5.3 反代配置模态框
使用 Vue 3 的 Teleport 和 Transition：
```vue
<Teleport to="body">
  <Transition name="modal">
    <div v-if="showProxyModal" class="modal-overlay">
      <div class="modal-content">
        <!-- 配置表单 -->
      </div>
    </div>
  </Transition>
</Teleport>
```

### 5.4 额度信息展示
使用分段进度条：
```vue
<div class="quota-bar">
  <div class="segment subscription" :style="{ width: subscriptionPercent + '%' }"></div>
  <div class="segment free" :style="{ width: freePercent + '%' }"></div>
  <div class="segment bonus" :style="{ width: bonusPercent + '%' }"></div>
</div>
```

## 8. 风险和缓解措施

### 8.1 API 结构变化风险
**风险**: Kiro API 响应结构可能变化，导致额度提取失败  
**缓解**: 
- 使用灵活的字段匹配逻辑
- 提供降级方案（显示总额度）
- 添加详细的错误日志

### 8.2 客户端启动失败风险
**风险**: 切换账号时客户端可能启动失败  
**缓解**:
- 提供清晰的错误提示
- 允许用户手动配置客户端路径
- 添加重试机制

### 8.3 配置数据丢失风险
**风险**: localStorage 清除导致配置丢失  
**缓解**:
- 提供配置导出/导入功能
- 定期备份配置数据
- 添加配置恢复提示
